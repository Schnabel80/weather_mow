# v0.4.2 — Ladezeit, Mäh-Status & Übersetzungs-Konsistenz

**Datum:** 2026-06-03
**Ziel-Release:** v0.4.2b1
**Status:** Design genehmigt

## Überblick

Drei Logik-Verbesserungen plus eine durchgängige Übersetzungs-Konsistenz:

1. **Aktives Mähen sichtbar machen** — kein irreführender Block-Grund während des Mähens
2. **Gelernte Ladezeit** (Zeit-zu-SoC) — selbstkalibrierende Laderate
3. **`next_mow_expected` aus Nässe + Dringlichkeit + Akku** — realistische nächste Mähzeit
4. **Übersetzungs-Konsistenz** — alle Entitäten + `block_reason`-States in HA-Sprache

---

## Hintergrund / Problem

Beobachtet am Live-System (Mäher mäht, Akku 66% < min 80%):

- `block_reason` zeigt `battery_low`, obwohl der Mäher **gerade mäht** — verwirrend.
- `next_mow_expected = 15:30` war **reine Nässe-Vorhersage**: Der Code geht bei jedem
  `start_now=False` (egal welcher Grund) in `_forecast_next_mow`, das nur die Rasennässe
  über 48h simuliert. Akku-Ladezeit kommt darin **nicht** vor. Da die Nässe schon 0 war,
  gab es `jetzt+1h + 30min Grace` zurück — weder Ladezeit noch sinnvoll.
- Die Anzeige mischt Sprachen: Binary-Sensor-**Zustände** zeigt HA-Core auf Deutsch
  („Nein"), `block_reason` zeigt rohe englische Keys, Entitäts-**Namen** sind teils
  englisch hartkodiert, teils via `translation_key` übersetzt.

## Abschnitt 1 — Aktives Mähen sichtbar machen

### Verhalten

- **Erkennung:** Mäher-Entity-State == `mowing` UND `stop_now` ist nicht gesetzt.
- **`block_reason`:** Neuer Wert `mowing_active` (englischer Roh-Key, konsistent mit
  `too_wet`, `battery_low` etc.). Wird angezeigt **statt** des sonst ermittelten Grundes,
  solange aktiv & gewollt gemäht wird.
- **`next_mow_expected`:** Während aktivem Mähen = `now` (Timestamp).
- **Wichtig:** Wenn `stop_now` gesetzt ist (Regen/Nässe während Mähen), bleibt der
  **echte** Grund sichtbar — ein nötiger Stopp darf nicht verschleiert werden.
- Ändert **nichts** an `start_now`/`stop_now`-Logik — rein Anzeige.

### Umsetzung

In `_async_update_data`, nachdem `block_reason`/`start_now`/`stop_now` final feststehen
und `next_mow_expected` berechnet wird:

```python
mower_state = self.hass.states.get(cfg.get(CONF_MOWER_ENTITY, ""))
is_mowing = mower_state is not None and mower_state.state == "mowing"
if is_mowing and not stop_now:
    block_reason = "mowing_active"
    next_mow_expected = now_local
```

## Abschnitt 2 — Gelernte Ladezeit

### Neues Modul `charging.py` (pure Python, HA-frei, wie `wetness.py`/`drying.py`)

Funktionen:

```python
DEFAULT_CHARGE_RATE_PCT_PER_MIN = 1.0   # Startwert bis erste valide Messung
CHARGE_RATE_MIN = 0.2                    # Klammer unten
CHARGE_RATE_MAX = 3.0                    # Klammer oben
FIRST_LEARN_MIN_RISE_PCT = 60.0          # Erstmessung erst ab diesem Anstieg
EMA_LEARN_MIN_RISE_PCT = 20.0            # Folgemessungen ab diesem Anstieg
EMA_ALPHA = 0.2                          # Glättung für Folgemessungen

def learn_charge_rate(
    current_rate: float,
    learned: bool,
    measured_rate: float,
    rise_pct: float,
) -> tuple[float, bool]:
    """Gibt (neue_rate, learned) zurück.

    - rise_pct < FIRST_LEARN_MIN_RISE_PCT und not learned → unverändert
    - rise_pct < EMA_LEARN_MIN_RISE_PCT und learned       → unverändert
    - not learned (Erstmessung, rise ≥ 60%)               → measured (α=1.0), learned=True
    - learned (rise ≥ 20%)                                → EMA 0.8·alt + 0.2·measured
    Ergebnis immer auf [CHARGE_RATE_MIN, CHARGE_RATE_MAX] geklammert.
    """

def minutes_to_target(battery_now: float, target_pct: float, rate: float) -> float:
    """max(0, (target - now) / rate). rate ≤ 0 → 0 (kein Schätzwert)."""
```

### Laderaten-Erkennung im Coordinator (pro 5-Min-Update)

Neue Instanz-Felder: `_charge_rate`, `_charge_learned`, `_charge_start_pct`,
`_charge_start_ts`.

Logik:
- **Ladephase-Start:** Mäher docked UND Akku steigt gegenüber `_prev_battery_pct` UND
  `_charge_start_ts is None` → `_charge_start_pct = prev`, `_charge_start_ts = now`.
- **Ladephase-Ende:** Akku fällt ODER Mäher verlässt Dock (`mowing`) ODER Akku == 100 →
  `rise = battery_now − _charge_start_pct`, `minutes = (now − start_ts)/60`.
  Wenn `minutes > 0` und `rise > 0`:
  `measured = rise / minutes`;
  `(_charge_rate, _charge_learned) = learn_charge_rate(_charge_rate, _charge_learned, measured, rise)`.
  Danach `_charge_start_ts = None`.

„Docked" = Mäher-State in {`docked`, `paused`} und Akku steigt; robuster: Akku steigt UND
State ≠ `mowing`. (Detail in Implementierung.)

### Persistenz

Neuer Store `STORAGE_KEY_CHARGE` = `weather_mow_<entry_id>_charge`:
```json
{"charge_rate_pct_per_min": 1.0, "learned": false}
```
- `_flush_storage` schreibt mit.
- `_load_storage` lädt mit Default 1.0 / learned=false; Klammer beim Laden.

### Sichtbarkeit

**Keine neue Entität.** Laderate nur als zusätzliche Spalte im Debug-CSV
(`charge_rate_pct_per_min`).

## Abschnitt 3 — `next_mow_expected` aus Nässe + Dringlichkeit + Akku

### Zweig-Priorität (ersetzt Block ab `_async_update_data` Zeile ~2127)

1. **Mäht gerade aktiv** → `now` *(Abschnitt 1)*
2. **`start_now`** → `now` *(unverändert)*
3. **`waiting_for_favorable`** mit `_last_drying_mm > 0` → bestehende Trocknungs-Schätzung
   *(unverändert)*
4. **Tagesziel erreicht** (`block_reason == "daily_target_reached"`) → nächster
   Mähfenster-Start am Folgetag.
5. **Sonst** → `max(dry_time, charge_time)`:
   - `dry_time = _forecast_next_mow(...)` — **mit adaptiver Schwelle** (s.u.)
   - `charge_time`: wenn `battery_pct < min_batt` → `now + minutes_to_target(battery_pct, min_batt, _charge_rate)`, sonst `now`
   - Ergebnis = **späterer** der beiden Zeitpunkte. Fällt `charge_time` außerhalb des
     Mähfensters, wird auf den nächsten Fenster-Start geschoben.

### Adaptive Schwelle im Forecast (Approach A — bestehendes binäres `urgency_high` spiegeln)

`_forecast_next_mow` bestimmt heute pro Stunde nur `grace_active_h = rain_next_3h == 0`.
Neu: pro simulierter Stunde dieselbe Schwellenwahl wie `_compute_decision`:

- **Zeitdruck** (`h_local >= target_end_dt` des jeweiligen Tages) → `mow_threshold_urgent`, keine Grace
- **Regen in den nächsten 3 Sim-Stunden** → `mow_threshold`, keine Grace
- **sonst** → `mow_threshold − FORECAST_DISCOUNT_MM`, Grace

`emergency_mow_active` und das Gras-Defizit-Kriterium sind Live-Zustände; im Forecast
wird die Zeitdruck-Komponente pro Stunde berechnet (sie ist zeitabhängig und damit
simulierbar). Hohe Priorität wirkt indirekt — kein gleitendes Priority-Threshold-Mapping
(bewusst nicht, Approach A).

### Konsequenz (Live-Beispiel)

Rasen 0mm trocken, Akku 66% < 80%, Rate 1%/min:
`dry_time = now`, `charge_time = now + (80−66)/1 = now + 14min`
→ `next_mow_expected ≈ now + 14 min` statt 15:30.

## Abschnitt 4 — Übersetzungs-Konsistenz

### Bestandsaufnahme

**Schon übersetzt** (`translation_key` + Eintrag in de/en): alle `number` (4),
`time` (1), `button` (2), `date` (1), `switch.debug_log`, `sensor.next_mow_expected`.

**Hartkodiert englisch** (25 Entitäten):
- **sensor (13):** wetness, priority, duration_today, duration_avg_3d, rain_last_1h,
  rain_weighted_12h, rain_today_total, rain_today_remaining, rain_tomorrow, solar_peak,
  dew_point, block_reason, grass_growth
- **binary_sensor (9):** allowed, start_now, stop_now, emergency_mow, raining,
  dew_present, brightness_ok, auto_resume_blocked, irrigation_active
- **switch (3):** enabled, emergency_mow, irrigation_active

### Umsetzung

- Jede der 25 Entitäten: `name=`/`_attr_name=` → `translation_key="<key>"`
- `de.json` + `en.json`: `entity.<platform>.<key>.name` für alle
  (en = bisherige englische Namen, de = deutsche Entsprechung)
- **`block_reason`:** zusätzlich `state`-Übersetzungen für alle 11 Werte

### `block_reason` State-Texte (de)

| Key | Deutsch |
|---|---|
| `mowing_active` *(neu)* | Mäht gerade |
| `mowing_allowed` | Mähen erlaubt |
| `too_wet` | Zu nass |
| `battery_low` | Akku zu niedrig |
| `waiting_for_favorable` | Wartet auf besseres Fenster |
| `daily_target_reached` | Tagesziel erreicht |
| `emergency_mow_tomorrow_rain` | Notmähen (Regen morgen) |
| `outside_time_window` | Außerhalb Mähfenster |
| `too_dark_hedgehog` | Zu dunkel (Igelschutz) |
| `too_hot` | Zu heiß |
| `disabled` | Deaktiviert |

(en: jeweils sprechende englische Entsprechung.)

## Tests

- **`charging.py`** (pure Python, eigene Datei `tests/test_charging.py`):
  - Erstmessung < 60% Anstieg → keine Änderung
  - Erstmessung ≥ 60% → ersetzt Default komplett (α=1), learned=True
  - Folgemessung < 20% → keine Änderung
  - Folgemessung ≥ 20% → EMA 0.2
  - Klammer 0.2–3.0 greift bei Aus­reißern
  - `minutes_to_target`: normal, schon über Ziel (=0), rate ≤ 0 (=0)
- **Coordinator:**
  - `mowing_active` gesetzt wenn Mäher mowing & nicht stop_now; echter Grund bleibt bei stop_now
  - `next_mow` = now während Mähen
  - `next_mow` = max(dry, charge) bei battery_low; charge-Zeit korrekt
  - adaptive Schwelle im Forecast (Zeitdruck-Stunde → urgent threshold)
  - Laderaten-Erkennung: Ladephase erkannt, gelernt, persistiert/geladen
- **Übersetzungs-Vollständigkeit:** Test iteriert alle `translation_key` der Plattformen
  und prüft Präsenz in de.json UND en.json (kein fehlender Key); block_reason-States
  vollständig.

## Nicht im Scope (YAGNI)

- Gleitendes Priority→Threshold-Mapping (Approach B) — bewusst verworfen.
- Konfigurierbare Laderate als Entität — nur intern + Debug-CSV.
- Mehrere Mäher / Multi-Instance-Besonderheiten.
